# 用户的个人约定

以下是用户的个人开发环境约定与工作流偏好。这是全局性的基础约定，适用于所有任务。

## 工具栈选取

- **不要使用 `rm`，而是使用 `trash`**。撰写用于分发给其他人的 shell 脚本时除外，撰写本机使用的 shell 脚本时依旧遵循此规则。
- C++ 项目使用 CMake 作为构建工具。
- 对于其他需要提供构建脚本的情况，使用 Justfile 替代 Makefile。
- Go 项目使用 goenv 指定版本，尽量使用 `go doc` 获取远端包的文档。
- Java 项目使用 jenv 指定版本。
- `pnpm` 用于安装作为工具的 JS 包。
- Python 项目管理使用 `uv`，尽量使用 `.venv`。
- JavaScript 项目管理使用 `bun`，跨项目使用的 JS 包使用 `pnpm` 管理。
- Java 项目管理使用 Maven。
- C++ 项目管理使用 Conan。
- 如果用户要求，使用 Nix 保证可复现性。
- 尽可能不要使用系统包管理器安装软件包。

## 目录安排

- 如果需要本地运行新的软件设施，先使用 `question` 工具请示用户，让用户做出明确决策。
- 如果用户允许，在 `$HOME/docker` 目录下创建新文件夹，并将开发用的容器放入 `tough-development-domain` 网络中。
- 如果需要验证脚本，使用 `cd $(mktemp -d)` 到 `/tmp` 目录下操作。

## 其他要求

- 使用 Makefile 时先阅读其内容，确保不会安装软件到用户家目录之外，避免污染环境。
- 尽量不要使用 Root 权限，除非用户明确做出相关决策。
- 撰写文档时尽量使用人类的书写习惯。
- **所有文字回答的结尾附带一个「喵」**。但是在编辑代码时不要这样干。
- 如果觉得当前工作可以抽离成模板，使用 `question` 工具请示用户，让用户授权你应用 skill-creator 工具将相应的工作抽离成 skill。
- **用户没有要求时，不允许在文档和文字输出中使用任何 emoji 字符。**
